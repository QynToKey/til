# Entity Relation Diagram （ER図）

## ER設計のロードマップ

### ① 概念モデル (Conceptual Model)

- エンティティ定義

  👉 *「なぜそのテーブルが必要なのか」を説明できるか❓*

- 責務整理

  👉 *「そのテーブルがやらないこと」は何か❓*

- リレーション整理

  👉 *「依存の流れ」を説明できるか？*

- 設計思想の明文化

  👉 *「このアプリの中心概念」は何か❓*

  👉 *設計で優先するものは❓ (柔軟性❓ 正規性❓ 実装容易性❓)*

⬇

### ② 論理モデル

- 主キー (PK)

  👉 *各テーブルの `PK` は❓*

- 外部キー

  👉 *外部キーの依存方向は❓*

- NOT NULL 制約

  👉 *そのカラムは「必須」か❓ 「任意」か❓*

- UNIQUE 制約

  👉 *そのデータが「一意」であることを、何によって保証するか❓*

- 依存関係

  👉 *「推移的依存」は存在するか❓ (多対多 の関係が生じているか❓)*

- 正規化確認

  👉 *「[第3正規形](https://school.runteq.jp/v3/books/2/chapter/14)」を満たしているか❓*

⬇

### ③ 物理モデル

- データ型確定

- index設計

  👉 *index を貼るべきカラムは❓*

- パフォーマンス観点

  👉 *集計はどこで行う❓*

- Rails migration 設計

  👉 *`foreign_key: true`, `null: false`, `default: false` の扱いは❓*

⬇

### ④ ER図（Mermaid）

- 上記を可視化する

  ```text
  言語化
    ↓
  可視化
    ↓

    例）
  ```

## `Mermaid` の記法

- **[Mermaid](https://mermaid.js.org/#/)** とは： フローチャート、シーケンス図、クラス図、ガントチャート、および git グラフを生成するための**マークダウン構文**

- 今回は `VS Code` に `拡張機能 Markdown Preview Enhanced` をインストールした。

  👉 *マークダウン記法で記述し、`Open Preview` で展開する*

### エンティティ定義

  ```typescript
  // 実装例
    erDiagram
      USERS {
          bigint id PK
          string email "UNIQUE"
      }
  ```

- テーブル名は慣習的に大文字で記述する
- `PK` = Primary Key
- `FK` = Foreign Key
- 「UNIQUE 制約」や「`boolean`」のデフォルト値などは、`""`(ダブルクォーテーション) を使って注釈として記す

### リレーション記号

  ```typescript
  // 実装例
  USERS ||--o{ LEARNING_THEMES
  ```

  👉 *1 user は 0以上の `learning_themes` を持つ*

  | 記号 | 意味 |
  | --- | --- |
  | `\|\|--\|\|` | 1 対 1 |
  | `\|\|--o{` | 1 対 多 （親は必須、子は0以上） |
  | `o\|--o{` | 0または1 対 多 （親は任意） |
  | `}o--o{` | 0以上 対 0以上 （多対多） |
  | `\|{--\|{` | 1以上 対 1以上 （両側とも最低1件は必須） |

  👉 *多対多のときは「中間テーブル」が必須*

  👉 *`|{--|{`(1以上 対 1以上) は業務ルールで担保することが多い*
