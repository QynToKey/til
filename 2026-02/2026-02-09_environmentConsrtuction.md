# 環境構築の tips と作法

## clone / fork / template の整理

### 今回の流れ

1. `Use this template` → 自分のリポジトリ作成（`fork`ではなく`template`を使用）
2. そのリポジトリを `git clone`

    👉 *`git init` は不要 (`git init` は 「Git 管理されていないディレクトリ」を新規に管理したいときだけ使う)*

### clone

- リモートリポジトリを**履歴を含めてローカルにコピー**する
- clone 後は `origin` が自動で設定される

### fork

- 他人のリポジトリを **自分のアカウントにコピー**

   👉 *`GitHub`の機能 (`Git`の機能ではない)*

- **元リポジトリとの親子関係**が GitHub 上に残る

### Use this template

- テンプレートから「完全に独立した新規リポジトリ」を作成
- 元リポジトリとの親子関係・履歴は**引き継がれない**
- 学習用途等で「元に戻す必要がない」場合に最適

#### クローンとフォークの違い

| 観点 | clone | fork |
| --- | --- | --- |
| 実行場所 | ローカル | リモート (GitHub 上) |
| 目的 | ローカルで作業する | 自分用にリポジトリを持つ |
| push 可否 | 権限次第 | 常に可能（自分のリポジトリ） |
| OSS 開発での利用 | △（閲覧・検証） | ◎（貢献） |

#### 使い分けのイメージ

| 目的 | Git操作 |
| --- | --- |
| OSS に貢献したい | fork |
| 課題・教材・雛形 | Use this template |
| チーム開発 *(push権限有)* | clone |
| 完全新規プロジェクト | `git init` |

## VS Code ターミナルの運用例

### 実務的なルール

- サーバーを起動しているターミナルは閉じない

    👉 *止めたいときは `Ctrl + C` で明示的に止める*

- 複数ターミナルに名前をつけて管理する

  - 構成例

    | ターミナル名 | 稼働させるコマンド例 |
    | --- | --- |
    | rails-server | `rails server` |
    | console | `rails c` |
    | work | `git status`, `bin/rails db:migrate` |
