# [HLWIL](https://github.com/QynToKey/HowLongWillItLast) : Pagy

## 背景

- 「学習ログ」が溜まってくると、同一ページで表示するのは無理が生じる
- 記録を見返す体験を豊かにするリファクタリングの一環として、ページネーションを導入したい。

## pagy vs kaminari

| 観点 | pagy | kaminari |
| --- | --- | --- |

## pagy の実装

### 1️⃣ `gem pagy` を追加

```ruby
# Gemfile 既存行の下に追加
gem "sorcery", "0.18.0"
gem "rails-i18n", "~> 8.1.0"
+ gem "pagy", "~> 9.0"
```

  ⬇️

```bash
docker compose run --rm web bundle install
```

📝 `--rm`： コマンド実行後にコンテナを自動で削除するオプション
`bundle install` のような一時的な作業用コンテナは実行後に残しておく必要がないため、--rm を付けてクリーンに保つのが慣習

### 2️⃣ イニシャライザーを作成 / 設定

> **Rails initializer** とは

アプリ起動時（サーバー起動・コマンド実行時）に**一度だけ実行される初期設定ファイル**。
`config/initializers/` 以下に置いたファイルはすべて Rails が自動で読み込む。

> **initializer** が必要な理由

- gem によっては `require` しただけでは動かず、**アプリ固有の設定を注入する場所**が必要。
- initializer はその「設定の差し込み口」として機能する。
- モデルや環境ごとの設定（`config/environments/`）より前に読み込まれるため、アプリ全体に確実に反映される。

```bash
touch config/initializers/pagy.rb
```

```ruby
require "pagy/extras/bootstrap"   # ① Bootstrap用ヘルパーを有効化

Pagy::DEFAULT[:items] = 20        # ② 1ページあたりの件数をデフォルト設定
```

📝 `require "pagy/extras/bootstrap"` ：
pagy 本体は軽量に作られており、Bootstrap 用のナビゲーション HTML を生成する機能はオプション扱い。
これを `require` することで `pagy_bootstrap_nav` ヘルパーが使えるようになる。

📝 `Pagy::DEFAULT[:items] = 20` ：
全コントローラー共通のデフォルト件数を設定する。
個別に変えたい場合は `pagy(records, items: 10)` のように上書きする。

### 3️⃣ `Pagy::Backend` モジュールを `ApplicationController` クラスに組み込む

- `Pagy::Backend` を `include` することで、`include` したクラスとその**サブクラスすべてに機能が継承**される。
- `ApplicationController` はすべてのコントローラーの親クラスなので、1か所に書くだけで `LearningRecordsController` を含む全コントローラーで `pagy()` が使えるようになる。

```ruby
class ApplicationController < ActionController::Base
+ include Pagy::Backend
  before_action :require_login
  # ...
```

📝 **モジュール** ： クラスに「機能のセット」を追加する仕組み。

| モジュール | 役割 | 組み込む場所 |
| --- | --- | --- |
| `Pagy::Backend` | データの取得・カウント処理 | `ApplicationController` |
| `Pagy::Frontend` | ナビゲーション HTML の生成 | `ApplicationHelper` |

```md
@pagy, @learning_records = pagy(records)
  ┃       ┗━━ ページ分割された件数
  ┗━━ ページ情報（現在ページ・総ページ数など）
```

内部では、渡された ActiveRecord Relation に対して

1. `COUNT` クエリで総件数を取得
2. `LIMIT` / `OFFSET` を付けて該当ページ分だけ取得

という2つのクエリを自動で発行する。

---

### 4️⃣ `ApplicationHelper` に `Pagy::Frontend` を追加

```ruby
# app/helpers/application_helper.rb
module ApplicationHelper
+ include Pagy::Frontend
end
```

| `Pagy::Backend` | `Pagy::Frontend` |
| --- | --- |
| データを扱う | 表示を扱う |

> 📝 `Pagy::Backend` をコントローラーに組み込むことで：

- **DB に問い合わせる**処理を担う
- `LIMIT` / `OFFSET` の計算、`COUNT` クエリの発行する
- 結果として「ページ情報オブジェクト `@pagy`」と「該当レコード」を返す

```ruby
@pagy, @learning_records = pagy(records)
```

> 📝 `Pagy::Frontend` をヘルパーに組み込むことで：

- **HTML を生成する**処理を担う
- `@pagy` のページ情報を受け取り、「前へ / 次へ / 1 2 3…」のナビゲーションを組み立てる
- Bootstrap 用・シンプル用など出力形式を切り替えられる

```ruby
<%== pagy_bootstrap_nav(@pagy) %>
```

> 📝 なぜ分かれているのか：

| pagy | MVC |
| --- | --- |
| `Backend` | **C**ontroller（データ処理） |
| `Frontend` | **V**iew（表示生成） |

- Rails の設計思想 **MVC** に対応している。
- 責務を分離することで、たとえば「API モードで HTML を出力しないコントローラー」には `Backend` だけ組み込む、という使い方もできる。

### 5️⃣ `LearningRecord` コントローラーを修正 （`index` アクション）

```ruby
# app/controllers/learning_records_controller.rb
  def index
    @learning_records = if params[:theme_id].present?
                          current_user.learning_records.where(learning_theme_id: @learning_theme.id).includes(:tags).order(study_date: :desc)
    else
                          current_user.learning_records.includes(:tags).order(study_date: :desc)
    end

    if params[:tag_id].present?
      @current_tag = current_user.tags.find(params[:tag_id])
-     @learning_records = @learning_records.joins(:tags).where(tags: { id: @current_tag.id })
+     @learning_records = @learning_records.joins(:tags).where(tags: { id: @current_tag.id }).distinct # ⬅️ 重複を排除するためにdistinctを追加
    elsif params[:date].present?
      @learning_records = @learning_records.where(study_date: params[:date])
    end

+   @pagy, @learning_records = pagy(@learning_records) # pagy を適用
  end
```

📝 `.distinct` ：
タグ JOIN 時に1件のレコードが複数行になる可能性があり、pagy の COUNT がずれるのを防ぐため。
